# M2 security-review carry-forward

Advisory findings from the security-auditor pass on F2.1+F2.2 (no BLOCKING
issues; the policy-engine core was confirmed sound). Each is dormant while
`jarvisd` wires `tools: None`, but MUST be addressed in the feature that first
activates the relevant path. Tracked here so a later session sees them.

| # | Finding | Fix in | Severity |
|---|---------|--------|----------|
| CF-1 | `Float` scalar in `canonical_form` was terminator-delimited, not length-prefixed. | **DONE in F2.2** (length-prefixed; test added). | advisory |
| CF-2 | `AuditSink::record` is fire-and-forget — cannot be written in the same transaction as the run's domain change (invariant #6, docs/06 §7). The port signature must let the audit event commit atomically with the checkpoint/outbox write, or be reworked so F2.4 can. | **PARTIAL in F2.4**: the security-critical grant lifecycle (`grant.minted`/`consumed`/`rejected`) is now written in the SAME tx as the grant row change inside `PgGrantStore` — atomic, via `audit::append`. The *orchestrator-emitted* audit path (`policy.approval_requested`, `approval.denied`, and R0/R1 `tool.*`) still flows through the fire-and-forget `AuditSink` port; unifying that with the checkpoint/outbox write lands with the live orchestrator wiring in **F2.5/F2.6**. **F2.6 Slice 2 note:** the new CF-4 `tool.executed`/`tool.failed`/`grant.mint_failed`/`approval.invalid_args` events still flow through the fire-and-forget `AuditSink` port, so a crash after a side effect but before the sink flushes leaves an unaudited effect — CF-4 makes closing this the F2.6 **Slice 3** `PgAuditSink` job (schedule explicitly, do not slip). **F2.6 Slice 3b — DURABILITY HALF DONE:** `jarvis_infra::audit_sink::PgAuditSink` now implements the `AuditSink` port and is wired into the live `ToolPlane`, so orchestrator-emitted events are hash-chain **persisted** (in the sink's own tx, via `audit::append`), not dropped. The failure path logs only `event_type` (a machine code), never the payload (invariant #5). **ATOMICITY HALF STILL OPEN:** the port is `record(&self, event) -> ()` (no tx thread, no error channel), so the append is not atomic with the run's checkpoint/outbox — a crash between the side effect and the sink commit still leaves an unaudited effect. Closing that needs the port to thread the caller's transaction (a domain/application port change, human-decision territory). CF-2 stays open for the atomicity half. | SHOULD-FIX (grant path + durability DONE; atomicity open) |
| CF-3 | Tool-result smuggling (docs/06 §5): `ToolResult.content` is folded verbatim into the next model prompt with no schema-validation, size-truncation, or control-char stripping. The `truncated` flag implies an upstream validator that does not yet exist. A result validator MUST enforce this — in the orchestrator/domain, not trusted per-executor — before a real `ToolExecutor` lands. | **DONE in F2.6 Slice 2**: `jarvis_domain::tools::sanitize_result_content` (pure) strips C0/C1/DEL control bytes except `\n`/`\t` and hard-caps length on a UTF-8 boundary; the orchestrator applies it in `replan_step` at the single choke point BEFORE folding into the prompt, ORing its `truncated` with the executor's. Domain property tests + an orchestrator wiring test. (web.fetch Z4 reuses it in **F2.8**.) | SHOULD-FIX (DONE) |
| CF-4 | Tool *execution* + result are not audited, only the policy *decision*. docs/06 §7 treats the side effect as the audited unit. Add an execution/result audit event when a real executor lands. | **DONE in F2.6 Slice 2**: `tool_step` records `tool.executed` on success and `tool.failed` on an execution error (payload = tool id only, never content — invariant #5); asserted in the auto- and R2-path orchestrator tests. | advisory (DONE) |
| CF-5 | `DataEgress` is defined but `evaluate` never consults it. External-egress tools must be gated before they are registered. | **RESOLVED in F2.8 Slice 3 — gate-before-registration (owner-approved).** The chosen disposition is *config-gated registration*, not consulting egress in `evaluate`: the external-egress `web.search`/`web.fetch` tools register ONLY when `[integrations.web_search]` is configured (`jarvisd::tools::register_web_tools`, wired in `main.rs` behind `if let Some(web) = &config.integrations.web_search`), so no ambient external-egress tool exists without explicit host consent. `DataEgress::External` is surfaced on the policy and on the approval card. R0+External still auto-authorises per call by design (the user directed the query); attaching *sensitive context* to an outbound request stays an NFR-02 context-assembler concern (F2.9 `Location` is sensitivity-labeled + only attached via `localize_query`). | advisory (RESOLVED) |
| CF-7 | `GrantBinding` (`jarvis-application::policy`) derives `Debug` over its raw `arguments: CanonicalValue`. No code debug-logs it today (no leak in F2.4), but a future `tracing` field on a `GrantBinding` would spill raw arguments into logs (invariant #5). Give it a manual `Debug` that renders `arguments` as their hash, or redact the field. | **DONE in F2.6 Slice 2**: `GrantBinding` now has a manual `Debug` that renders `arguments` as `<redacted>` (the argument *hash* is what the audit trail records). | advisory (DONE) |
| CF-6 | The F2.3 grant ports are infallible (`GrantMinter::mint -> ExecutionGrant`, `GrantValidator::validate -> Result<(), GrantError>`), so an infra/DB fault in `PgGrantStore` has no error channel and `.expect()`-panics. This is FAIL-SAFE (a panicked mint/validate authorizes nothing — no side effect), but not graceful: it aborts the task instead of routing to `RunState::Failed`. Give the ports an error arm and handle it in the orchestrator when the live tool path is wired. | **DONE (mint) in F2.6 Slice 2**: `GrantMinter::mint -> Result<ExecutionGrant, GrantMintError>`; `PgGrantStore::mint` returns `Err` (before `commit`, so the tx rolls back — no partial grant) and the orchestrator routes it to `Failed` with a stable non-sensitive detail, never executing. Tested with a `FailingGrantMinter`. `GrantValidator::validate`'s own error arm (its DB faults still `.expect`) remains a follow-up. | advisory (mint DONE) |
| CF-8 | **The WS hub broadcasts every domain event to every authenticated connection with no per-user/session filter** (`jarvisd/src/ws.rs`: single `broadcast::Sender`, `handle_socket` subscribes unfiltered, `replay_since` replays all outbox rows). F2.5's `approval.requested` card is now the **highest-value payload** on that channel — it carries `exactEffect` + the real `proposedArguments` (e.g. a `message.send` recipient + body) + the `approvalId` (a decision oracle). Pre-existing infra (F2.5 does not touch `ws.rs`; `message.created` already fans out the same way); dormant only because the deployment is loopback-only single-user (multi-user is M7-gated, `api.rs`). Scope hub delivery AND `replay_since` by the connection's `UserId`/session (filter by run ownership) before M7. | **M7** (before multi-user) | SHOULD-FIX (dormant: single-user) |
| CF-9 | **Edited approval arguments are not schema-validated at resolve time.** `JarvisApprovalGate::resolve` lifts arbitrary human JSON (`editedArguments`) straight into the bound `ApprovalOutcome::Approved { arguments }` via `json_to_canonical`. Safe in F2.5 because binding does not execute — the grant binds `tool_id`/`target_resource`/`tool_version` from the *proposal* (an edit cannot redirect to another tool/resource), and validation+consumption happens at the executor boundary. But the edit path bypasses the proposal's `policy::evaluate` input-schema check. Validate edited args against the tool input schema before execution. | **DONE in F2.6 Slice 2**: `ToolExecutor::validate_args` (default-Ok; a schema-bearing tool overrides) is now called by `approval_step` on the human's *approved* arguments BEFORE the grant is minted — a malformed edit is rejected at binding time (`approval.invalid_args` audit, run `Failed`), never minting a grant nor executing. Tested at the orchestrator level. Native-adapter overrides land when the live ToolStack wires (Slice 3). | advisory (DONE) |
| CF-10 | **`fs.read` containment has two residual holes, both requiring a concurrent local writer** (out of the model-supplied-path threat this R0 read defends; security-auditor F2.6 Slice 1). (a) TOCTOU: a directory on the canonical path can be swapped for a symlink between `resolve`'s `canonicalize` and `tokio::fs::File::open`, redirecting the open outside root; (b) a hardlink placed inside root to an outside file reads the target while `starts_with(root)` passes (`canonicalize` does not resolve hardlinks). Dormant: this tier ships **no write tool** that could plant either, single local user. Airtight fix is `openat2(RESOLVE_BENEATH\|RESOLVE_NO_SYMLINKS)` (or per-component `O_NOFOLLOW`). Documented as a known limit in `fs_read.rs::resolve`. Harden when the first fs-write tool or the live ToolStack lands. | **F2.6 Slice 3 / first write tool** | advisory (dormant: no write tool) |
| CF-12 | **`Debug`-over-raw-arguments siblings of CF-7** (rust-reviewer + security-auditor F2.6 Slice 2). CF-7 gave `GrantBinding` a redacted `Debug`, but `ToolProposal`/`ToolInvocation` (`jarvis-domain::tools`), `ApprovalRequest`, and `ApprovalOutcome::Approved` (`jarvis-application::policy`) still `#[derive(Debug)]` over raw `CanonicalValue` arguments. No code `tracing`/`{:?}`-logs them today (grepped — no leak), so this is latent, not a live leak; but a future tracing field on any of them would spill raw args (invariant #5). Give each the same redacted `Debug`. | **DONE in F2.7 Slice 3**: `ToolProposal`/`ToolInvocation` (`jarvis-domain::tools`) and `ApprovalRequest`/`ApprovalOutcome` (`jarvis-application::policy`) now have manual `Debug` impls that render `arguments`/`exact_effect`/`proposed_arguments` as `<redacted>` (mirroring CF-7 `GrantBinding`), keeping tool_id/risk/egress for correlation. **Wire twins also covered** (security-auditor F2.7 Slice 3): `ApprovalCardDto`/`ApprovalDecisionDto` (`jarvis-contracts::approvals`) carry the same `exact_effect`/`proposed_arguments`/`edited_arguments` and now have manual redacted `Debug` too (`Serialize` still carries the real values to the UI). Redaction tests assert the secret values never appear in `Debug` yet remain in `Serialize`. Not sensitive, correctly left deriving `Debug`: `PolicyContext` (ULIDs + capability `Scope` names). Still-open sibling: **`ToolResult.content`** derives `Debug` over tool output (e.g. `fs.read` file bytes) — lower risk (already control-stripped; it is output not args) but a `?result` tracing field would surface it; fold into CF-13's F2.8 pass if the result path grows tracing fields. | advisory (DONE) |
| CF-13 | **Result sanitizer does not strip Unicode bidi-override / zero-width format chars** (`U+202E`, `U+200B`, …) (rust-reviewer + security-auditor F2.6 Slice 2). `sanitize_result_content` strips C0/C1/DEL as CF-3 specified, but visual-spoofing format chars survive into the timeline/prompt. Out of CF-3's stated scope (control bytes + length), so not a gap against the requirement — harden when **F2.8** `web.fetch` reuses the validator on richer HTML-derived text. | **DONE in F2.8 Slice 1**: `sanitize_result_content` now also strips bidi embeddings/overrides/isolates + explicit marks (`U+200E/200F`, `U+202A–202E`, `U+2066–2069`) and zero-width chars (`U+200B/200C/200D`, `U+FEFF`) via `is_bidi_or_zero_width` — closing URL/text spoofing on the first tool (`web.search`) that renders provider-controlled URLs. Domain test asserts an RLO-spoofed domain + zero-width joiners are stripped. Applies to every result path (orchestrator CF-3, mcp_host, web). | advisory (DONE) |
| CF-11 | **`ToolPolicy::timeout` is not enforced and R2 grant-presence is not asserted at the executor boundary** (security-auditor F2.6 Slice 1). `orchestrator::tool_step` races `executor.execute` only against the cancel token, not against `policy.timeout`, so a FIFO/special file or a slow trickle under the read cap blocks until the user cancels. Separately, `tool_step` validates a grant only under `if let Some(grant)` — the `RunState` machine structurally guarantees an R2 call carries one, but there is no explicit `policy.requires_grant() ⇒ grant present` belt at the boundary. Both land with the live ToolStack wiring: honor `policy.timeout` in `tool_step`, and add the grant-presence assertion (defence in depth on invariant #1). | **PARTIAL in F2.6 Slice 3a**: (a) grant-presence belt is **DONE + live** — `tool_step` fetches `policy_of` and fails the run (no `resolve`, no `execute`) if `requires_grant() && grant.is_none()`, backed by the unit-tested pure `grant_presence_ok` predicate; it is defence in depth *on top of* the cryptographic `grant_validator.validate` that still runs. (b) the timeout is **built but DORMANT**: `jarvis_adapters::tools::timeout::TimeoutExecutor` (host-owned `Duration`, drop-on-elapse → `ToolError::Timeout`, `validate_args` forwarded; `start_paused` tests) exists but has **no production caller yet**. **Slice 3b MUST wrap every registered executor** with `TimeoutExecutor::wrap(executor, policy.timeout)` at ToolStack registration — a tool registered without the wrap silently loses its deadline, so the wrap must be applied uniformly at the single registration site (docs/06 §8 gate 3: "every R2/R3 tool has a timeout" is not satisfiable until then). **DONE in F2.6 Slice 3b:** `jarvisd::tools::build_registry` routes every `register(...)` through `wrap_with_timeout`, applying the host-owned `ToolPolicy.timeout` uniformly at the single registration site; a policy-less descriptor is refused by `ToolRegistry::register` (`MissingPolicy`), never run unbounded — so a registered tool is necessarily a timeout-wrapped tool (no bypass). Both reviewers confirmed the wrap is uniform and un-lengthenable by the tool. | DONE (belt + timeout) |
| CF-14 | **Timeout/cancel gives no execution atomicity for a future mutating tool** (security-auditor F2.6 Slice 3a). `TimeoutExecutor` (and the orchestrator's cancel) drop the inner `execute` future at an await point. For the current read-only `fs.read` and the demo `message.send` (sends nothing) a mid-flight drop leaves no partial effect. When a real mutating executor lands, a drop on elapse could leave a partial external side effect — covered by the registered **compensation + idempotency key** (docs/06 §5 replay/duplicate-mutation control), never by the timeout decorator. Fold into the CF-10 "first write tool" hardening. | **first write tool** | advisory (dormant: no mutating tool) |
| CF-15 | **A requeued or crash-recovered run loses its originating tool authority** (rust-reviewer + security-auditor F2.6 Slice 3b). The run's `PolicyContext` is not carried through the durable queue / restart recovery, so `poll_provider_health` (degraded requeue) and `recover_unfinished_runs` (crash recovery) re-spawn with `policy = None` → `tools: None` (`main.rs`). This **fails closed** (invariant #1: no ambient authority for an unattributable run) and is **inert in M2** — a queued run failed at the model call *before* any tool proposal, and the Claude CLI adapter proposes no tools (ADR-004). But once MCP tools are live (F2.7), a run that proposed a tool and is then requeued/recovered will silently diverge (tools present on attempt 1, absent on retry). Decide deliberately in F2.7: either persist the run's `PolicyContext` with the durable run so recovery re-attributes it, or explicitly *fail* such runs rather than silently degrade. | **RESOLVED in F2.7 Slice 3 — fail closed.** Deliberate decision (docs/CLAUDE.md ambiguity rule: stricter security + simpler operational interpretation): a requeued/crash-recovered run is **not** silently re-granted its originating tool authority. It re-spawns with `policy = None` → `tools: None` (`RunEngine::should_wire_tools`, truth-table tested) — no ambient authority for a run whose device session may be gone (invariant #1). *Re-attributing* a recovered run (persist + **re-validate** `PolicyContext` against the still-current device/scopes) is intentionally NOT done here: it is a larger design (durable-run schema change + revalidation semantics) that would weaken the fail-closed default, so it is deferred to its own future design/ADR if ever wanted. Inert in M2 regardless (Claude CLI proposes no tools, ADR-004). | advisory (RESOLVED: fail closed by design) |
