# M2 security-review carry-forward

Advisory findings from the security-auditor pass on F2.1+F2.2 (no BLOCKING
issues; the policy-engine core was confirmed sound). Each is dormant while
`jarvisd` wires `tools: None`, but MUST be addressed in the feature that first
activates the relevant path. Tracked here so a later session sees them.

| # | Finding | Fix in | Severity |
|---|---------|--------|----------|
| CF-1 | `Float` scalar in `canonical_form` was terminator-delimited, not length-prefixed. | **DONE in F2.2** (length-prefixed; test added). | advisory |
| CF-2 | `AuditSink::record` is fire-and-forget тАФ cannot be written in the same transaction as the run's domain change (invariant #6, docs/06 ┬з7). The port signature must let the audit event commit atomically with the checkpoint/outbox write, or be reworked so F2.4 can. | **PARTIAL in F2.4**: the security-critical grant lifecycle (`grant.minted`/`consumed`/`rejected`) is now written in the SAME tx as the grant row change inside `PgGrantStore` тАФ atomic, via `audit::append`. The *orchestrator-emitted* audit path (`policy.approval_requested`, `approval.denied`, and R0/R1 `tool.*`) still flows through the fire-and-forget `AuditSink` port; unifying that with the checkpoint/outbox write lands with the live orchestrator wiring in **F2.5/F2.6**. | SHOULD-FIX (grant path DONE) |
| CF-3 | Tool-result smuggling (docs/06 ┬з5): `ToolResult.content` is folded verbatim into the next model prompt with no schema-validation, size-truncation, or control-char stripping. The `truncated` flag implies an upstream validator that does not yet exist. A result validator MUST enforce this тАФ in the orchestrator/domain, not trusted per-executor тАФ before a real `ToolExecutor` lands. | **F2.6** (native tools) and **F2.8** (web.fetch Z4) | SHOULD-FIX before any executor |
| CF-4 | Tool *execution* + result are not audited, only the policy *decision*. docs/06 ┬з7 treats the side effect as the audited unit. Add an execution/result audit event when a real executor lands. | **F2.6** | advisory |
| CF-5 | `DataEgress` is defined but `evaluate` never consults it. External-egress tools must be gated before they are registered. | **F2.8/F2.9** | advisory |
| CF-7 | `GrantBinding` (`jarvis-application::policy`) derives `Debug` over its raw `arguments: CanonicalValue`. No code debug-logs it today (no leak in F2.4), but a future `tracing` field on a `GrantBinding` would spill raw arguments into logs (invariant #5). Give it a manual `Debug` that renders `arguments` as their hash, or redact the field. | **F2.5/F2.6** (when the approval/exec path logs) | advisory |
| CF-6 | The F2.3 grant ports are infallible (`GrantMinter::mint -> ExecutionGrant`, `GrantValidator::validate -> Result<(), GrantError>`), so an infra/DB fault in `PgGrantStore` has no error channel and `.expect()`-panics. This is FAIL-SAFE (a panicked mint/validate authorizes nothing тАФ no side effect), but not graceful: it aborts the task instead of routing to `RunState::Failed`. Give the ports an error arm and handle it in the orchestrator when the live tool path is wired. | **F2.6** | advisory |
| CF-8 | **The WS hub broadcasts every domain event to every authenticated connection with no per-user/session filter** (`jarvisd/src/ws.rs`: single `broadcast::Sender`, `handle_socket` subscribes unfiltered, `replay_since` replays all outbox rows). F2.5's `approval.requested` card is now the **highest-value payload** on that channel тАФ it carries `exactEffect` + the real `proposedArguments` (e.g. a `message.send` recipient + body) + the `approvalId` (a decision oracle). Pre-existing infra (F2.5 does not touch `ws.rs`; `message.created` already fans out the same way); dormant only because the deployment is loopback-only single-user (multi-user is M7-gated, `api.rs`). Scope hub delivery AND `replay_since` by the connection's `UserId`/session (filter by run ownership) before M7. | **M7** (before multi-user) | SHOULD-FIX (dormant: single-user) |
| CF-9 | **Edited approval arguments are not schema-validated at resolve time.** `JarvisApprovalGate::resolve` lifts arbitrary human JSON (`editedArguments`) straight into the bound `ApprovalOutcome::Approved { arguments }` via `json_to_canonical`. Safe in F2.5 because binding does not execute тАФ the grant binds `tool_id`/`target_resource`/`tool_version` from the *proposal* (an edit cannot redirect to another tool/resource), and validation+consumption happens at the executor boundary. But the edit path bypasses the proposal's `policy::evaluate` input-schema check. Validate edited args against the tool input schema before execution. | **F2.6** | advisory (execution unwired) |
