## Why F1.7 is complex

The feature requires coordination between several layers:
1. **Adapter** (jarvis-adapters): classify errors as quota/auth/network/timeout
2. **Application** (jarvis-application): maintain provider health state, manage queue transitions
3. **Host** (jarvisd): spawn health polling task, wire queue into RunEngine, add endpoints
4. **Contracts** (jarvis-contracts): wire ProviderHealthChanged/RunQueued events (already done)

The single-flight queue in F1.6 ensures `Invoking` state only happens once; F1.7 extends this to *queue* runs when the provider is down and *dequeue* when it recovers.
