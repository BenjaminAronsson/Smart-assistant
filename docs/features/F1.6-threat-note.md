# F1.6 Threat Note — Claude CLI adapter with single-flight queue and health detection

## Feature scope
- Claude CLI adapter: spawn `claude` binary with streaming JSON, cancellation, restart recovery
- Single-flight queue: prevent concurrent model runs (security/quota enforcement)
- Health detection: classify failures (quota exhausted, auth failure, rate limit, network, timeout, malformed output)
- Exit hints: surface reset windows and actionable next steps from health signals
- Contract: `ProviderModel` trait with `invoke()` returning `ModelResponse` (streaming or one-shot)
- Integration: wire adapter into `RunEngine` as concrete `ProviderModel` impl

## Attack surface

### 1. Model output injection (Invariant 1)
**Threat:** Model generates malicious JSON that causes side effects through unsafe parsing or re-interpretation.
**Specifics:** 
- Claude CLI output contains embedded tool invocations or code execution directives
- Stream-JSON parsing fails to validate event types, allowing unknown keys to propagate
- Assistant message content is not sanitized before display

**Mitigations:**
- Stream-JSON parser validates against `ProviderEvent` enum; unknown event types logged with warning, discarded
- Assistant content is never interpreted as executable (always rendered as text/code blocks, never as code to run)
- Tool invocation must route through policy engine (invariant 1 boundary), never directly from model output
- All domain events pass through orchestrator state machine; model output is inert until state transition approves it

### 2. Concurrent/replay attacks (Invariant 4, IND-5)
**Threat:** Multiple runs invoke the model simultaneously, consuming quota faster than intended or causing billing/auth conflicts.
**Specifics:**
- Restart recovery re-drives unfinished runs while a new question is in flight → duplicate model invocations
- No serialization barrier between user input and model spawn

**Mitigations:**
- **Single-flight queue:** RunEngine holds a mutex-guarded semaphore (capacity 1); spawn() acquires it, releases on terminal state
- Restart recovery skips runs already in the active registry (checked before re-driving)
- Only one `orchestrator::Run` transitions through `Invoking` state at a time (state machine guarantees exhaustive transitions)

### 3. Quota exhaustion DoS (IND-4, resource constraint)
**Threat:** Quota-limited user hammers the system with questions; quota is spent before they see degraded-mode warning.
**Specifics:**
- Quota check happens after model invocation starts (too late)
- Health detection doesn't classify quota-exhausted state; user sees generic error
- No backoff or queue when quota is low

**Mitigations:**
- Quota polling happens before spawn (in health probe, separate from invoke path)
- Quota-exhausted is classified as `ProviderHealth::Unavailable(RetryAfter)` from health detector
- RunEngine checks health before spawning; if unavailable, run queues in deterministic-fallback mode (F1.7)
- Health signal surfaces reset window via domain event; UI shows "quota resets at X" with countdown

### 4. Provider crash/hang (IND-6)
**Threat:** Claude CLI hangs indefinitely or crashes mid-response; user sees spinning wheel, server holds resources.
**Specifics:**
- No timeout on subprocess read loops
- Crash leaves zombie process or dangling pipes
- Cancellation token not wired to subprocess termination

**Mitigations:**
- Subprocess spawned with stdout pipe; read loop uses `tokio::time::timeout(PROVIDER_IDLE_TIMEOUT, socket.read_frame())`
- Idle timeout triggers cancellation of run (propagates to subprocess via `process.kill()` on next poll)
- Span guard holds `Child` process handle; drop kills it; graceful shutdown joins with timeout
- Subprocess is spawned fresh per run (no pooling, simpler lifecycle)

### 5. Malformed/incomplete JSON (invariant 1)
**Threat:** Claude CLI outputs incomplete or broken JSON; parser crashes or silently skips events.
**Specifics:**
- Stream ends mid-event (connection drop) → incomplete final JSON line
- Event object is missing required fields (`type`, `index`, `delta`)
- Deserialization error propagates as panic or silent retry

**Mitigations:**
- `tokio::io::BufReader` with line-by-line parsing; partial final line is logged as warning and discarded
- `serde_json::from_str` errors are mapped to `ParseError` domain error; run completes with `Outcome::ParseError`
- Known event types are validated; unknown types log a warning, don't error
- Minimum viable event validation: `type` and `delta` required for content events

### 6. Process resource leaks (NFR-16)
**Threat:** Many spawned Claude processes accumulate without cleanup.
**Specifics:**
- Early exit paths don't close pipes or kill subprocess
- Graceful shutdown doesn't wait for subprocesses to finish

**Mitigations:**
- `Child` process held in span-scoped variable; drop() kills it
- Pipes (stdout, stderr) are explicitly closed
- Graceful shutdown calls `engine.drain()` with timeout before exit (main.rs already does this)
- TaskTracker per run ensures subprocess task is joined before run is considered terminal

## Trust boundary

- **Subprocess stdin/stdout:** Data from `claude` binary is untrusted wire format; must deserialize and validate
- **Subprocess env/args:** Never pass user input directly; sanitize via structured args struct or safe defaults
- **Health signals:** Quota reset hints, error messages from Claude come from model's JSON; never execute them, only display
- **Run recovery:** Session ID and run ID are trusted (ULID, already persisted); re-drive with session's latest user message only

## Invariant checklist

1. ✅ Text never grants authority: Model output is inert until orchestrator processes it; spawn is guarded by state machine
2. ✅ State machine owns the loop: RunEngine is decision point; all model invocations from `Received` → `Invoking`; no ambient tool calls
3. ✅ Domain stays pure: `ProviderModel` trait is in `jarvis-application::ports`; adapter in `jarvis-adapters` depends on application, not vice versa
4. ✅ Everything cancellable: Subprocess spawned with `CancellationToken`; `Child` handle is scoped; graceful shutdown times out and kills
5. ✅ No secrets in prompts/logs: Subprocess args use safe enums (no user prompt in argv); auth token never logged; secrets resolver at adapter boundary only
6. ✅ Append-only audit: Model health state changes are logged as `health.changed` events; never retroactively rewritten
7. ✅ Determinism: Streaming provider is deterministic when seeded with fixtures; golden traces use `FakeModel` for reproducible replay

- [ ] Claude CLI adapter spawns with streaming JSON parsing

- [ ] Health detection classifies quota/auth/rate-limit/network/timeout/parse errors
