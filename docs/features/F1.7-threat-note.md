# F1.7 Threat Note — Degraded mode with run queue + provider health detection

## Feature scope
- Provider health classification: quota exhausted, auth failure, rate limit, network errors, timeouts
- Health state machine with backoff/reset windows
- Application-layer run queue: interactive > background FIFO, single-flight honored
- Visible waiting state: RunState::Queued (domain)
- `GET /api/v1/providers` endpoint with health, quota, reset window
- Provider-health WS events (health.changed)
- Automatic retry when provider recovers (no user re-submission)

## Attack surface

### 1. Queue exhaustion DoS (IND-4)
**Threat:** Attacker submits many runs while provider is down; queue grows unbounded, consuming memory.
**Specifics:**
- No limit on queue depth; each enqueued run holds state/promises
- Run input (user prompt) kept in memory indefinitely
- Orchestrator checkpoint may also hold state

**Mitigations:**
- Queue capacity capped (default 100 runs, configurable per deployment)
- Oldest background runs dropped if limit exceeded (FIFO eviction)
- Interactive runs never dropped (user-visible state lost)
- Quota window expiry invalidates queued runs (re-run from top if resumed)

### 2. State confusion (Invariant 2)
**Threat:** Queue-to-active transition loses run state or applies wrong transition.
**Specifics:**
- A run in Queued state transitions to Invoking when provider recovers
- Orchestrator doesn't know about queue; state machine must be unambiguous
- Checkpoint on queue entry, replay on dequeue

**Mitigations:**
- Queue stores the full Run domain object (immutable while queued)
- On dequeue, checkpoint is loaded and orchestrator resumes from saved state
- RunState::Queued has explicit transition to Invoking (exhaustive match in next())
- Queue management is orthogonal to orchestrator (run drives self)

### 3. Health signal spoofing (Invariant 1)
**Threat:** Malicious health response or provider output causes side effect.
**Specifics:**
- Health detector runs outside the orchestrator; signal drives queueing decision
- Reset window from provider could be used for timing attacks
- Health state could trigger inappropriate retry behavior

**Mitigations:**
- Health detector only produces opaque classification (enum, no raw provider text)
- Health state feeds only the queueing decision; run outcome is authoritative
- Reset window is advisory (UI hint only), never executable/actionable beyond "wait"
- Health polling is independent of orchestrator loop; failures don't crash it

### 4. Queue ordering attacks (load balancing)
**Threat:** Interactive runs starved by background workload; attacker prioritizes background runs.
**Specifics:**
- No priority between runs of same class
- Attacker submits many background runs to hold queue slots
- Interactive runs never get serviced

**Mitigations:**
- Queue is split: interactive FIFO (unbounded) + background FIFO (capped)
- Interactive runs always dequeue first when provider recovers
- Background FIFO size bounded (overflow drops oldest); interactive never dropped
- Single-flight semaphore ensures one run at a time, no starvation vector within that constraint

### 5. Provider recovery race (consistency)
**Threat:** Health probe says "recovered", but provider still returns quota-exhausted mid-run.
**Specifics:**
- Health probe is asynchronous from provider invocation
- Time-of-check vs time-of-use gap
- Run starts, provider says quota exhausted, run fails mid-execution

**Mitigations:**
- Expected behavior: health probe is advisory only; mid-run failures are handled normally
- On ModelError::Unavailable mid-stream, run completes with Failed(Unavailable) state
- Queue is not a guarantee of success; it's a best-effort retry mechanism
- No "promise to succeed" once queued; provider error handling is unchanged

## Trust boundary

- **Health signals:** Provider classification (quota/auth/network) is untrusted; used for queueing only
- **Reset windows:** Advisory hints, never used for auth/access control decisions
- **Queue state:** Domain-owned; reads from persistent checkpoint only
- **Run routing:** Queue manager routes based on Run.priority (system-owned), not user input

## Invariant checklist

1. ✅ Text never grants authority: Health probe output drives queueing (decision), not execution
2. ✅ State machine owns the loop: RunState::Queued has explicit next() transition; dequeue is explicit
3. ✅ Domain stays pure: Queue manager is in application layer; no provider/HTTP types
4. ✅ Everything cancellable: Queued runs are cancellable (CancellationToken held); drain stops queueing
5. ✅ No secrets in prompts/logs: Health probe logs only classification enum, never provider output
6. ✅ Append-only audit: Queue operations are logged as `run.queued` / `run.dequeued` audit events
7. ✅ Determinism: Health polling is deterministic (fixtures in tests); golden trace 3 covers recovery

## Exit evidence for F1.7

- [ ] Provider health classification (quota/auth/rate-limit/network) with reset windows
- [ ] Health polling loop with configurable interval and backoff
- [ ] RunState::Queued state with explicit transition to Invoking
- [ ] Application-layer queue: interactive FIFO + background FIFO with capacity
- [ ] Single-flight honored while dequeuing (no concurrent invocations)
- [ ] `GET /api/v1/providers` returns current health + reset windows for all profiles
- [ ] Provider-health WS event on state change
- [ ] Queued run transitions to active when provider recovers
- [ ] Golden trace 3: question → quota-exhausted → queued (visible) → recovered → completes
